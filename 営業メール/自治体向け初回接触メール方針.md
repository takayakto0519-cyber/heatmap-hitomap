# 自治体向け初回接触メールの方針（ペルソナC：関係人口の「入口はあるが出口がない」自治体担当者）

作成：2026-07-22。標茶町（北海道）からの返信（丁重なお断り：「既存の体制で運用できており、新規導入を検討する時期ではない」）を教訓に策定。
municipality_profiles.email_draft を今後AIが生成・更新する際は、必ずこの方針に沿うこと。

## このペルソナの正体

関係人口・移住定住の「入口施策」（お試し住宅、ふるさと納税、体験ツアー、SMOUT登録など）は、ほぼ全ての自治体がすでに何かしら持っている。彼らに欠けているのは入口ではなく、**一度きりの来訪を「もう一度関わりたい」に育てる出口**。ヒトマップはここに刺さる。

「新しいシステムの導入」として伝えると、標茶町のように「今は時期じゃない」で弾かれる。**置き換えではなく、今ある取り組みに重ねるだけ**、という一点を必ず伝えること。

## 書き方の4原則

1. **相手の既存施策を先に肯定する**。否定・比較（「〜が足りていません」等）から入らない。まず「拝見しました」「素晴らしい取り組みだと感じました」で始める。
2. **「置き換えではなく重ねるだけ」を明言する一文を必ず入れる**。例：「すでにある〇〇の仕組みを変えていただく必要はありません。そこに、訪れた人の"感じたこと"を可視化する一枚を重ねるだけです。」
3. **CTAは「15分の面談」より軽くする**。第一候補は「まずは資料だけでもご覧いただけないか」。面談を求める場合も「ご都合が合えば」「情報交換程度で」という温度に抑える。予約リンク（https://hitomap.com/schedule）を併記し、返信のハードルを下げる。
4. **署名は「ヒトマップ　加藤貴也」に統一する**。過去のドラフトで「ヒトマップ運営一同」等の揺れがあった（丸亀市版）。文体・結びも含め、この方針書のテンプレートに揃える。

## 適用範囲
- `municipality_profiles.email_draft` を新規生成・更新するすべてのAIエージェントセッション（`gov-proposal-custom`スキルが提案書本体を作る際も、初回接触メールはこの方針に従う）
- 既存の下書きで未送信のものは、順次この方針で書き直す（全83件を一度に書き直さず、スコア上位から着手）

## 出典ルール（2026-07-22追記：新規開拓31件の一斉ファクトチェックで、7件に捏造疑い・事実誤認・リンク切れが発見されたことを教訓に追加）

**`evidence_summary` に書く具体的な数字・固有名詞・制度名・年号は、`source_links` のいずれかのページに直接記載されている内容だけを書く。** 一般知識・推測・「たぶんこうだろう」で補完しない。裏付けが取れない具体情報は、書かない（骨子だけの一般的な表現に留める）か、そもそもそのリードを見送る。

新規リードを1件登録するたびに、以下を自己チェックしてから保存すること：
1. `evidence_summary` の各文について、`source_links` のどのURLの記載に基づくかを自分の中で説明できるか（説明できない文は削除する）
2. 数字（年数・人数・金額・年号）は特に厳密に。桁や年数を「それらしく」丸めたり足したりしない
3. `website_url` は実在し、当該自治体の公式または準公式サイトであることを確認したか
4. 被災地・係争中の話題など、状況が変化しやすい対象は特に慎重に、時制（「〜している」進行形か「〜していた」過去形か）を確認する

保存時は `fact_check_status` を必ず設定する：
- 上記1〜4を確認した上で登録した場合 → `verified`
- 確認したが一部に不安が残る場合 → `unverified`（デフォルト。理由を`fact_check_note`に一言添える）
- 過去に登録済みの行を見直して問題を見つけた場合 → `flagged`（`fact_check_note`に何が問題かを明記）

`fact_check_status` が `verified` でない下書きは、送信キューのボタンが自動的に無効化される（`app/api/admin/municipality-profiles/[id]/send/route.ts` 等のサーバー側ガード）。会長がダッシュボードで最終確認して「事実確認済みにする」を押すか、AIエージェントが出典と突き合わせた上で `verified` を明示的にセットするまで、誤った情報が入ったメールは物理的に送信できない。
